# Daily Bloom PRD

## 1. 产品概述

Daily Bloom 是一个面向个人日常任务管理的轻量级 Todo 应用。产品强调清新、平静、低负担的使用体验，帮助用户快速完成账号注册登录、查看今日任务、追踪完成进度，并通过底部弹窗快速新增任务。

当前产品已经具备前端视觉原型和基础全栈能力：React + Tailwind 前端、Better Auth 邮箱密码认证、tRPC Todo 接口、PostgreSQL + Drizzle 数据存储。

## 2. 产品目标

- 让用户打开应用后直接进入登录/注册流程。
- 让用户能快速创建、查看、完成和删除今日任务。
- 用「Daily Goal」进度卡即时反馈当天任务完成度。
- 用柔和的视觉风格降低任务管理工具的压力感。
- 为后续扩展分类、截止时间、日历、专注模式、个人中心打好信息架构基础。

## 3. 目标用户

- 需要轻量任务管理的个人用户。
- 偏好简洁、清新、移动端优先体验的用户。
- 关注日常习惯、工作任务、生活事项和健康安排的人群。

## 4. 当前页面与路由

### 4.1 默认入口

- 路由：`/`
- 行为：自动重定向到 `/login`
- 目的：用户打开 `http://localhost:3001` 后默认进入登录/注册页。

### 4.2 登录/注册页

- 路由：`/login`
- 默认状态：展示注册表单。
- 可切换状态：
  - 注册表单：姓名、邮箱、密码。
  - 登录表单：邮箱、密码。
- 成功后行为：跳转到 `/dashboard`。
- 错误反馈：通过 toast 展示认证错误。

### 4.3 Todo 任务页

- 路由：`/todos`
- 页面模块：
  - 顶部品牌栏：Daily Bloom、用户名称、设置按钮。
  - 欢迎区：展示问候语和进度鼓励文案。
  - Daily Goal 进度卡：展示完成百分比和完成数量。
  - Today's Tasks 列表：展示任务、完成状态、删除操作。
  - 空状态：没有任务时提示新增第一条任务。
  - 浮动新增按钮：打开新增任务底部弹窗。
  - 底部导航：Tasks、Calendar、Focus、Profile。

### 4.4 新增任务弹窗

- 触发：点击 `/todos` 页面右下角新增按钮。
- 展示形式：底部弹窗。
- 字段：
  - 任务名称。
  - 分类：Work、Personal、Home、Health。
  - 截止时间选项：今天、明天、选择时间。
- 当前提交行为：仅提交任务名称到现有 Todo API。
- 当前限制：分类和截止时间暂未落库。

## 5. 核心功能需求

### 5.1 用户注册

- 用户可以输入姓名、邮箱、密码注册账号。
- 邮箱必须符合邮箱格式。
- 密码不少于 8 位。
- 姓名不少于 2 个字符。
- 注册成功后跳转到 dashboard。
- 注册失败时显示错误提示。

### 5.2 用户登录

- 用户可以输入邮箱和密码登录。
- 邮箱必须符合邮箱格式。
- 密码不少于 8 位。
- 登录成功后跳转到 dashboard。
- 登录失败时显示错误提示。

### 5.3 查看任务

- 用户进入 `/todos` 后拉取任务列表。
- 加载中显示骨架屏。
- 无任务时显示空状态。
- 有任务时按列表展示任务标题、元信息和完成状态。

### 5.4 新增任务

- 用户点击浮动新增按钮后打开底部弹窗。
- 用户输入任务名称后可以提交。
- 空任务名称不可提交。
- 提交中按钮显示保存状态。
- 提交成功后关闭弹窗并刷新任务列表。

### 5.5 完成/取消完成任务

- 用户点击任务左侧勾选按钮切换完成状态。
- 完成后的任务标题展示删除线和弱化样式。
- 切换成功后刷新任务列表和 Daily Goal 进度。

### 5.6 删除任务

- 用户点击任务右侧删除按钮删除任务。
- 删除成功后刷新任务列表和 Daily Goal 进度。

### 5.7 今日进度

- 进度百分比 = 已完成任务数 / 总任务数。
- 总任务数为 0 时，进度为 0%。
- 进度卡展示：
  - 百分比，例如 `75% Completed`。
  - 计数，例如 `6 of 8 tasks`。
  - 可视化进度条。

## 6. 非功能需求

### 6.1 视觉风格

- 风格关键词：清新、轻盈、柔和、自然、低压力。
- 主色：薄荷绿。
- 辅助色：淡天空蓝、浅灰、柔和白。
- 避免高对比黑色、大面积沉重暗色、强烈装饰元素。
- 重点页面应移动端优先，同时在桌面端保持居中和最大宽度约束。

### 6.2 可用性

- 主要操作按钮必须明显可见。
- 新增任务必须可在一个底部弹窗内完成。
- 空状态必须明确告诉用户下一步。
- 加载状态必须避免页面突然跳变。

### 6.3 可访问性

- 表单字段必须有 label。
- 图标按钮必须有 `aria-label`。
- 图片必须有 alt 文案。
- 交互元素使用 button/input 等语义元素。

### 6.4 性能

- 页面首屏避免不必要的大型依赖。
- 图片必须设置宽高，降低 CLS 风险。
- Todo 列表加载使用骨架屏反馈。

## 7. 数据模型

### 7.1 当前 Todo 模型

当前数据库表只支持基础 Todo：

```ts
type Todo = {
  id: number;
  text: string;
  completed: boolean;
};
```

### 7.2 建议扩展 Todo 模型

为了完整支持当前 UI，后续建议扩展为：

```ts
type Todo = {
  id: number;
  userId: string;
  title: string;
  completed: boolean;
  category: "Work" | "Personal" | "Home" | "Health";
  dueAt: Date | null;
  createdAt: Date;
  updatedAt: Date;
};
```

## 8. API 需求

### 8.1 当前 API

- `todo.getAll`
- `todo.create`
- `todo.toggle`
- `todo.delete`

### 8.2 后续建议

- Todo API 应改为 protected procedure，只返回当前登录用户的数据。
- `todo.create` 支持 `title`、`category`、`dueAt`。
- `todo.update` 支持编辑标题、分类、截止时间。
- `todo.getToday` 支持按日期筛选今日任务。
- `todo.getStats` 支持返回每日完成率、连续完成天数等统计。

## 9. 当前限制

- Todo 数据暂未绑定用户，当前任务列表是全局数据。
- 分类和截止时间只是 UI 状态，提交时暂未写入数据库。
- 底部导航中的 Calendar、Focus、Profile 尚未实现对应页面。
- 设置按钮尚未实现功能。
- Dashboard 仍是基础认证示例页，不是 Daily Bloom 风格页面。

## 10. 后续迭代建议

### P0

- Todo 绑定当前登录用户。
- 新增任务支持分类和截止时间落库。
- 登录成功后跳转到 `/todos`，而不是示例 dashboard。
- 保护 `/todos` 路由，未登录用户跳转 `/login`。

### P1

- 支持编辑任务。
- 支持按分类筛选任务。
- 支持今日/明日/自定义日期任务筛选。
- 将顶部用户名称替换为真实 session 用户名。
- 完成 Profile 页面和退出登录入口。

### P2

- Calendar 页面展示按日期聚合的任务。
- Focus 页面提供专注计时和当前任务选择。
- 任务统计：每日完成率、连续完成天数、分类占比。
- 支持暗色模式下的 Daily Bloom 主题适配。

## 11. 验收标准

- 打开 `/` 会进入 `/login`。
- 用户可以在登录和注册表单之间切换。
- 注册/登录表单有基础校验和错误反馈。
- `/todos` 可以展示任务列表、空状态和加载状态。
- 用户可以新增、完成/取消完成、删除任务。
- 新增任务弹窗可以打开和关闭。
- Daily Goal 进度随任务完成状态变化。
- 页面符合 Daily Bloom 清新柔和的视觉方向。
